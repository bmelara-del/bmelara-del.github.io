Technical Documentation Page

Getting Started

This page is a quick technical reference for a clean, repeatable workflow using your terminal and Git.

The goal is to keep projects organized, commits meaningful, and fixes easy to trace.

If you’re working on GitHub Pages, you’ll typically edit files locally, commit changes, and push to your repository.

  • Install Git and confirm it works
  • Choose a code editor you like
  • Make sure you can open a terminal
git --version

Terminal Basics

The terminal is just a text-based way to navigate folders and run commands.

You’ll use it to move around your project, inspect files, and run Git commands.

  • pwd shows your current folder
  • ls lists files
  • cd changes directories

Example navigation flow:

pwd ls cd projects/my-site

Git Fundamentals

Git tracks changes in your project over time using commits (snapshots).

A good habit is to commit small chunks of work with clear messages.

The most common loop is: check status → add changes → commit → push.

  • git status shows what changed
  • git add . stages changes
  • git commit -m "message" saves a snapshot
git status git add . git commit -m "Add technical docs layout"

Branching Strategy

Branches let you experiment without breaking your main code. Think of them as parallel timelines.

A simple approach: keep main stable and do work in a feature branch.

  • Create a new branch for each feature
  • Commit as you go
  • Merge back into main when done

Typical branch workflow:

git checkout -b feature/navbar git commit -m "Style navbar"

Troubleshooting

When something goes wrong, don’t panic—use the tools Git gives you to inspect what happened.

If you’re lost, start with git status and read it slowly. It usually tells you what to do next.

  • If you staged the wrong files, use git restore --staged
  • If a merge is messy, pause and check git diff
  • If you need the commit history, use git log --oneline

Helpful inspection commands:

git log --oneline git diff
